## Why

快照与账户相关表单仍大量用手输日期与英文枚举值，用户难以快速选对日期，也无法按「某一天」从日历进入编辑；大事记金额缺少明确的收支语义。账户类型、归属、大事记分类等选项写死在后端校验与前端常量中，无法按家庭习惯在设置中维护。需要在不推翻现有数据模型的前提下，补齐**日历式导航与可配置枚举**，并顺带改善相关页面的选择与展示体验。

## What Changes

- **日历入口（探索结论 C）**：提供按日期查看/进入快照编辑的 UI（例如日历或日期选择 + 跳转）：若该日已有快照则进入编辑或详情，若无则引导新建并预填该日。
- **设置中的枚举维护**：为账户类型、账户归属、大事记分类等提供**每用户（或每租户）可配置**的选项列表（展示标签 + 稳定存储键），API 与校验从固定 `Set` 改为读取配置；历史数据中已存的键在展示时通过配置映射为标签，未知键需降级展示。
- **日期控件统一**：快照等需 `YYYY-MM-DD` 的字段改用日期选择器（Naive `n-date-picker` 等），减少手输错误。
- **大事记收支语义**：在表单与详情中明确区分收入/支出（例如符号、文案、或「类型」字段），与金额正负规则一致并在规格中写清。
- **UI  polish（范围内）**：与上述页面直接相关的间距、表格列标签中文化、空状态；不做全站重设计。

**BREAKING（潜在）**

- 若枚举从「固定集合」变为「数据库配置」，对外 JSON 若仍使用英文 `key` 则对客户端兼容；若改为仅允许配置内键，需定义**默认种子数据**与迁移策略，避免旧数据被拒绝。提案要求：**默认种子与当前硬编码枚举等价**，以降低破坏性。

## Capabilities

### New Capabilities

- `settings-api`: 用户级（或应用级）枚举/选项配置的 CRUD 与读取（账户类型、归属、大事记分类等），含默认值与校验规则。
- `settings-page`: 设置页：维护上述选项列表（增删改排序、标签文案），入口自主导航。

### Modified Capabilities

- `account-api`: 创建/更新账户时 `type`/`owner` 的合法值由设置配置决定，而非硬编码 `Set`；列表/详情响应可附带展示用标签（或由前端用配置映射）。
- `snapshot-api` / `snapshot-pages`: 日期选择器；日历或按日导航至编辑/新建；大事记金额/收支展示规则。
- `event-api` / `event-page`: 大事记分类合法值来自配置；统计页分类展示用标签。
- `vue-frontend`: 全局导航增加「设置」；枚举选项从 API 拉取并缓存；与日历相关的共享组件或路由约定。

## Impact

**后端**：新增设置相关表或 JSON 列、Flyway 迁移、`SettingsService`、调整 `AccountValidation`/`EventValidation` 或替换为配置驱动校验；必要时提供启动/首次访问时的默认种子数据。

**前端**：新设置页与 API 模块；`AccountsPage`、`SnapshotFormPage`、`SnapshotDetailPage`、`EventStatsPage` 等改为使用配置与中文化标签；快照列表或新增「日历/按日」视图路由。

**数据**：现有 SQLite 中已存储的 `type`/`owner`/`category` 字符串必须在配置中存在或提供兼容策略（自动插入默认配置项或只读展示「未知」）。
